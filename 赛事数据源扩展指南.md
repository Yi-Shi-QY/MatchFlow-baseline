# Match Data Source Extension Guide

This document is kept for compatibility with previous naming.

## Objective

Ingest third-party football data into `match-data-server` and expose normalized payloads for MatchFlow.

## Recommended Pattern

1. Build a scheduled ingestion script (cron or worker).
2. Map upstream teams/matches to local schema.
3. Upsert teams via `POST /admin/teams`.
4. Upsert matches via `POST /admin/matches`.
5. Keep odds and stats in normalized JSON shape expected by the app.

## Client Integration

Set in app settings:
- `matchDataServerUrl`
- `matchDataApiKey`

UI fetches matches through `src/services/matchData.ts`.

## Declarative Source Registry (New)

The start-analysis UI is now driven by a declarative registry:
- `src/services/dataSources.ts`

Each source entry defines:
- Source card metadata (`id`, `labelKey`, `descriptionKey`, `icon`, `cardSpan`)
- Availability/default rules (`isAvailable`, `defaultSelected`)
- Payload assembly rules (`applyToData`, `removeFromData`)
- Form schema (`formSections`, `fields`)

Current built-in source IDs:
- `fundamental`
- `market`
- `custom`

`src/pages/MatchDetail.tsx` no longer hardcodes source cards or form blocks.
It renders source cards and editable form sections by iterating the registry.

## SourceContext Contract

When user edits source selection, the app assembles `editableData` and injects:

```json
{
  "sourceContext": {
    "origin": "server-db | server-mock | local | imported",
    "selectedSources": {
      "fundamental": true,
      "market": false,
      "custom": true
    },
    "selectedSourceIds": ["fundamental", "custom"],
    "capabilities": {
      "hasFundamental": true,
      "hasStats": true,
      "hasOdds": false,
      "hasCustom": true
    },
    "matchStatus": "upcoming | live | finished | unknown"
  }
}
```

`src/services/ai.ts` uses these signals for deterministic planning-route selection
(template vs autonomous path and allowed agents).

## How To Add a New Source

1. Add a new source entry in `ANALYSIS_DATA_SOURCES` inside `src/services/dataSources.ts`.
2. Define:
   - selection defaults
   - payload apply/remove behavior
   - form sections and fields
3. Add i18n labels used by `labelKey`, `descriptionKey`, and field keys.
4. Optionally extend `buildSourceCapabilities` if planner routing needs a new capability signal.
5. Run:
   - `npm run lint`
   - `npm run build`

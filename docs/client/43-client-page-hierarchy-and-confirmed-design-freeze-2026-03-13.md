## 1. 文档目的

这份文档用于冻结当前已经确认完成的页面层级关系、一级信息架构和部分页面设计结论，便于后续继续细化页面时不丢失已经达成一致的决定。

它重点回答 4 个问题：

1. 当前产品的页面层级应该如何组织。
2. 一级页面之间的职责边界如何划分。
3. 设置、记忆及其二级页面的关系如何落地。
4. 哪些设计已经足够稳定，哪些仍然留待后续补充。

## 2. 当前产品定义

当前产品定义为：

`一个对话驱动的个人分析操作台。`

当前对外表达为：

- 足球分析与自动化研究助手
- 面向持续协作、任务闭环和长期上下文积累的个人助手

它不是：

- 通用 AI 平台
- dashboard 优先的资讯工具
- 只做单次问答的轻量聊天产品

## 3. 页面层级总览

本轮已经确认的页面层级如下。

```text
移动端应用壳层
├─ 一级页：对话中心
├─ 一级页：任务中心
├─ 一级页：分析与数据
├─ 一级页：历史
├─ 一级页：记忆
└─ 一级页：设置
   ├─ 二级页：连接与数据
   └─ 二级页：高级与诊断
      └─ 深层独立页：扩展管理

独立详情页
└─ 主题详情 / 分析执行页
```

其中：

- `对话中心`、`任务中心`、`分析与数据`、`历史`、`记忆`、`设置` 是一级工作空间。
- `连接与数据` 和 `高级与诊断` 只作为从 `设置` 进入的二级页，不进入侧栏一级导航。
- `扩展管理` 作为比 `高级与诊断` 更深一层的独立页保留。
- `主题详情 / 分析执行页` 是内容详情页，不属于一级工作空间切换项。

## 4. 移动端壳层与导航顺序

### 4.1 壳层结构

移动端壳层固定为：

- 顶部状态条
- 左上角抽屉侧栏
- 主内容区

本轮已确认：

- 顶部状态条放在顶部，不做底部状态栏。
- 一级导航采用左上角抽屉，不使用底部 Tab。
- 顶部状态条只显示压缩状态，不承担大段信息展示。

### 4.2 一级导航顺序

一级导航按以下顺序组织：

1. 对话中心
2. 任务中心
3. 分析与数据
4. 历史
5. 记忆
6. 设置

确认原则：

- 首页必须是对话中心，不做 dashboard 首页。
- 一级导航只保留“工作空间”级入口。
- 二级设置页和技术型页面不进入一级导航。

## 5. 一级页面职责冻结

本节只记录已经确认完成的职责边界，不展开未细化的卡片级规范。

### 5.1 对话中心

对话中心是产品首页，也是默认工作起点。

本轮已确认：

- 首页必须是简单清晰的对话模式。
- 系统可接收文字、语音和多模态输入。
- 系统可以直接回复，也可以先生成任务卡等待用户确认执行。
- 当存在未完成事项时，首页优先引导“继续协作”。
- 当上一次流程已经闭环时，首页只轻量展示上次摘要，然后把注意力还给新输入。
- 系统可以在定时触发、任务完成或遇到问题时主动回到对话中追问或汇报。
- 首页支持建议回复或快捷建议，但它们用于补全输入，不应抢占对话主流程。

对话中心首页进一步冻结为两种主态：

- `继续优先`
  - 当存在未完成事项、待确认任务、待回复问题、执行中事项或异常待处理事项时进入该状态。
  - 第一屏优先展示继续区，再进入消息流。
- `新输入优先`
  - 当上一次流程已经闭环且当前没有挂起事项时进入该状态。
  - 第一屏只轻量展示上次摘要，然后让输入重新成为主焦点。

首页的页面构成原则已经确认：

- 顶部状态条常驻。
- 消息流是主舞台。
- 底部输入区长期可见。
- 建议回复和快捷建议只做辅助，不替代用户输入。
- 任务确认、进度反馈、异常处理都尽量回到消息流中完成，而不是拆成独立流程页。

本轮已确认，首页第一屏固定骨架为：

1. 顶部状态条
2. 继续区或上次摘要区
3. 主消息流
4. 建议回复 / 快捷建议
5. 底部输入区

确认原则：

- 顶部状态条常驻，不抢占主舞台。
- `继续区` 与 `上次摘要区` 二选一，根据当前是否存在挂起事项切换。
- 主消息流始终是首页主舞台。
- 建议回复与快捷建议只做辅助输入，不直接替用户发送。
- 底部输入区长期可见，保证用户随时可以打断并发起新意图。

本轮已确认，首页 `继续区` 最多展示 1 到 3 张高优先级卡，不固定为单卡。

确认原则：

- 首页是协作入口，不是单事项详情页。
- 当同时存在多个需要继续处理的高优先级事项时，允许并列展示，但数量上限保持在 3 张以内。
- `继续区` 只承接最高优先级事项，不替代完整任务列表。

本轮已确认，首页 `继续区` 的默认优先级排序为：

1. 待确认任务卡
2. 待回复澄清问题
3. 异常待处理
4. 可恢复的执行中事项或未完成主题

确认原则：

- 优先展示直接阻塞当前协作链路的事项。
- 先处理需要用户确认或回复的节点。
- `可恢复主题` 虽然重要，但优先级低于直接阻塞项。

### 5.2 任务中心

任务中心已经确认定位为：

`当前所有待推进事项的处理台`

本轮已确认：

- 它不是技术自动化页。
- 它不以运行日志或内部状态为核心。
- 页面分区固定为：
  - 待我处理
  - 已安排
  - 执行中
  - 最近完成
- 顶部允许存在压缩概览，但不堆叠技术指标。

任务中心第一屏的组织原则已经确认：

- 先给出简短概览，让用户知道当前待处理量。
- 最优先区域必须是 `待我处理`，因为这是最需要用户决策的部分。
- `已安排` 用于承接已经进入系统但尚未完成的任务。
- `执行中` 用于查看正在推进的事项，而不是展示内部执行日志。
- `最近完成` 作为从任务流回到结果流的桥接区域。

本轮已确认，任务中心第一屏实际展示顺序固定为：

1. 顶部概览卡
2. `待我处理`
3. `执行中`
4. `已安排`
5. `最近完成`

任务中心的用户心智已经固定为：

- 这是“待办处理台”。
- 不是“自动化控制面板”。
- 不是“运行状态监控页”。

本轮已确认，顶部概览卡只展示 4 个压缩数字：

- 待我处理
- 执行中
- 已安排
- 最近完成

本轮已确认，概览卡不承载以下内容：

- 成功率
- 平均运行时长
- 自动化统计报表
- 其他监控台式指标

本轮已确认，`待我处理` 只承接真正需要用户动作的事项，固定收纳以下三类：

- 待确认执行
- 待补充信息
- 异常待决策

已经自动排队、但当前不需要用户介入的事项，不进入 `待我处理`，而进入 `执行中` 或 `已安排`。

本轮已确认，`待我处理卡` 的基础结构为：

- 标题
- 任务类型标签
- 为什么现在需要你处理
- 截止时间或建议处理时间
- 一个主操作
- 一个次操作

本轮已确认，`待我处理卡` 的主操作直接使用具体动作命名，不使用泛化的 `处理` 文案，例如：

- 待确认执行：`确认执行`
- 待补充信息：`继续补充`
- 异常待决策：`处理异常`

本轮已确认，`待我处理卡` 的通用次操作为：

- `稍后处理`

本轮已确认，`执行中卡` 的基础结构为：

- 标题
- 当前阶段
- 最近更新时间
- 一句进展说明
- 主操作：`查看进展`

本轮已确认：

- `执行中卡` 用于查看任务当前推进到哪一步。
- 它不承担运行控制台职责。
- 列表卡上不放 `暂停`、`终止` 等强控制按钮。

本轮已确认，`已安排卡` 的基础结构为：

- 标题
- 安排方式
  - 定时
  - 周期
  - 稍后执行
- 下次执行时间
- 一句执行说明
- 主操作：`查看安排`

本轮已确认：

- `已安排卡` 用于承接已经进入系统、但当前尚未执行或等待下一次触发的任务。
- 修改时间、停用规则、调整周期等操作进入详情页完成。
- 列表卡上不堆叠多按钮，避免任务中心变成规则编辑台。

本轮已确认，`最近完成卡` 的基础结构为：

- 标题
- 完成时间
- 一句结果摘要
- 主操作：`查看结果`

本轮已确认：

- `最近完成` 是从任务流返回结果流的区域。
- `最近完成卡` 以查看结果为第一动作，而不是再次操作任务。
- 如需继续跟进或再次执行，应进入结果详情页后再处理。

### 5.3 分析与数据

分析与数据已经确认定位为：

`可分析对象入口 + 数据可用性工作区`

本轮已确认：

- 它不是单纯的数据源设置页。
- 它需要承接“当前可分析对象”和“数据是否可用”这两类信息。
- 它可以吸收现有实现里与 live feed、可分析对象卡片、数据刷新相关的内容。

本轮已确认该页的第一性职责是：

- 帮用户判断现在有哪些对象可以继续分析。
- 帮用户判断当前数据是否可用、是否已更新。
- 把“分析对象入口”和“数据可用性”放在同一个工作空间中，而不是拆成纯浏览页和纯设置页。

本轮也已经确认：

- 该页不承担系统设置职责。
- 数据连接配置应从这里剥离，放入 `设置 -> 连接与数据`。
- 该页更偏业务工作台，而不是系统配置台。

本轮已确认，`分析与数据` 第一屏展示顺序固定为：

1. 顶部状态摘要
2. 当前可分析对象
3. 数据可用性
4. 最近同步 / 最近更新

本轮已确认：

- 该页的第一性问题是“现在能分析什么”。
- 数据状态服务于分析对象，而不是反过来成为页面主角。
- 最近同步与最近更新属于辅助信息，放在首屏靠后位置即可。

本轮已确认，`当前可分析对象` 区采用主题卡组织，而不是按原始数据流组织。

主题卡至少承载：

- 对象或主题名称
- 当前状态
- 简短数据可用性提示
- 主操作：`进入分析`

确认原则：

- 该区首先回答“我现在可以点开分析什么”。
- 不要求用户先理解底层数据流结构。
- 主题卡需要能和详情分析页、任务页、历史页形成一致的对象心智。

本轮已确认，`数据可用性` 区首版采用单张状态摘要卡，不采用长列表。

状态摘要卡至少展示：

- 当前数据状态
- 最近更新时间
- 是否存在异常
- 主操作：`查看详情`

确认原则：

- `数据可用性` 是辅助区，不压过 `当前可分析对象`。
- 用户只需要快速判断当前数据是否可用、是否异常、是否需要进一步查看。
- 连接与配置细节不在本页展开，而进入 `设置 -> 连接与数据`。

### 5.4 历史

历史已经确认是独立一级页。

本轮已确认：

- 第一屏优先展示最近完成的结果。
- 其次展示可继续内容。
- 再往下展示已保存主题。
- 历史页不承担对话入口和任务调度职责。

历史页的角色已经确认：

- 它是长期结果和既往主题的回看空间。
- 它是从“本次协作”过渡到“长期沉淀”的页面。
- 它承担结果复用和重新打开旧主题的职责，但不承担当前任务处理职责。

本轮已确认，`历史` 第一屏展示顺序固定为：

1. 顶部轻量摘要
2. 最近完成
3. 可继续内容
4. 已保存主题

确认原则：

- 用户进入历史页时，先看最近产出的结果。
- 其次再看仍可恢复、可继续的内容。
- 长期保存的主题作为稳定沉淀，放在其后。

本轮已确认，`历史页最近完成卡` 的基础结构为：

- 标题
- 完成时间
- 一句结果摘要
- 对象或主题标签
- 主操作：`查看结果`

本轮已确认：

- `历史页最近完成卡` 以结果浏览为第一动作。
- 它是结果入口，不是任务操作入口。
- 继续跟进或再次执行应进入结果详情页后再处理。

本轮已确认，`可继续内容卡` 的基础结构为：

- 标题
- 上次停留时间
- 一句可继续原因
- 主操作：`继续此主题`

本轮已确认：

- `可继续内容` 指向过去打开过、当前仍可恢复或继续推进的主题。
- 它不是“待我处理”任务卡的另一种写法。
- 历史页中的继续行为是恢复旧主题，而不是处理当前待办。

本轮已确认，`已保存主题卡` 的基础结构为：

- 标题
- 主题标签或对象标签
- 最近保存时间
- 一句摘要
- 主操作：`打开主题`

本轮已确认：

- `已保存主题` 是长期保留的对象入口。
- 它不等同于“可继续内容”。
- `已保存主题` 更偏稳定沉淀，而不是恢复执行链路。

### 5.5 记忆

记忆已经确认是独立一级页，与设置分离。

本轮已确认：

- 设置页不承载记忆内容编辑。
- 记忆内容必须可见、可解释、可编辑。
- 记忆以文档化形式存储和展示，内容形态可视作带元信息的 markdown 记录。
- 首版不提供手动新建记忆。
- 首版支持浏览、预览、编辑、启用、停用、确认候选、删除、合并重复项。

记忆页的产品角色已经确认：

- 它是用户可以直接审视系统长期认知的页面。
- 它负责承接用户偏好、系统推断习惯和每日摘要。
- 任何会影响未来推荐和默认行为的记忆，都必须先在这里可见。

### 5.6 设置

设置已经确认定位为：

`系统行为控制中心`

本轮已确认：

- 设置页不再是技术配置大杂烩。
- 设置页不再保留“基础模式 / 高级模式”双模式切换。
- 设置页采用正式产品语气，不暴露研发阶段说明和内部实现概念。

## 6. 设置体系冻结

这是本轮已经讨论闭环的部分，也是后续开发应直接依照执行的结构。

### 6.1 设置首页结构

设置首页固定为以下顺序：

1. 顶部轻量总览卡
2. 通用
3. 执行与提醒
4. 记忆与推荐
5. 连接与数据
6. 底部 `高级与诊断` 入口

### 6.2 顶部轻量总览卡

本轮已确认：

- 设置首页保留顶部轻量总览卡。
- 总览卡只展示状态，不直接承载复杂配置。
- 总览卡至少包含：
  - 当前系统状态
  - 异常或待处理数量
  - `AI`、`数据`、`记忆` 三个状态标签
- 总览卡中的状态标签支持直接跳转：
  - `AI`、`数据` 跳到 `连接与数据`
  - `记忆` 跳到 `记忆` 页

### 6.3 通用

`通用` 分区固定包含：

- 语言
- 外观
- 默认分析领域

这些设置属于正式设置首页可直接修改的内容。

### 6.4 执行与提醒

`执行与提醒` 分区固定包含：

- 自动执行
- 后台运行
- 通知相关状态

本轮已确认：

- 对用户展示正式产品语义，不暴露宿主、心跳、队列等内部运行术语。
- 这一区是行为控制，不是运行日志页。

### 6.5 记忆与推荐

`记忆与推荐` 分区只管理策略，不管理记忆正文。

本轮已确认：

- 该分区顶部保留一张摘要卡，显示：
  - 已启用数量
  - 待确认数量
  - 今日摘要数量
- 该分区至少包含以下策略项：
  - 是否记录长期偏好
  - 推断出的习惯是否先确认再使用
  - 是否生成每日摘要
  - 是否显示建议回复与快捷建议
- 该分区提供明确入口，进入独立 `记忆` 页。

### 6.6 连接与数据

`连接与数据` 在设置首页中只显示状态摘要，不直接展开所有配置表单。

本轮已确认：

- 它作为从设置进入的二级页存在。
- 不进入一级导航。
- 在设置首页上只承担摘要与入口职责。

### 6.7 高级与诊断

本轮已确认：

- `高级与诊断` 是独立二级页。
- 它作为设置底部独立入口存在。
- 它不进入一级导航。
- 它保留正式产品可理解的高级配置、检查、同步和维护能力。
- 开发联调专用的内部选项不应在正式版中显示。

## 7. 设置交互规则冻结

### 7.1 生效方式

本轮已确认：

- 设置项采用即时生效。
- 不再要求用户做统一保存。
- 页面内可通过轻量反馈提示更新成功。

### 7.2 顶部右侧按钮

本轮已确认：

- 设置页顶部右侧按钮保留。
- 按钮文案从 `保存` 改为 `完成`。
- 该按钮只承担结束当前设置流程的作用，不承担保存语义。

## 8. 连接与数据二级页冻结

`连接与数据` 是当前版本里可直接修改 AI 和数据源配置的正式二级页。

### 8.1 页面结构

页面固定为以下顺序：

1. 顶部统一状态卡
2. AI 服务
3. 数据源

### 8.2 顶部统一状态卡

本轮已确认：

- 页面顶部保留统一状态卡。
- 状态卡至少显示：
  - AI 状态
  - 数据源状态
  - 最近检查时间
  - 重新检查按钮

### 8.3 AI 服务

本轮已确认：

- AI 服务分区在当前版本直接展开，不做折叠。
- 当前版本直接向用户暴露并允许修改：
  - provider
  - model
  - base URL
  - API Key
- 页面内允许执行连接检查。
- 本版本先不为了未来账号体系增加额外复杂抽象。

### 8.4 数据源

本轮已确认：

- 数据源分区在当前版本直接展开，不做折叠。
- 当前版本直接向用户暴露并允许修改：
  - 数据服务地址
  - API Key
- 页面内允许执行数据连接检查。

### 8.5 与未来账号体系的关系

本轮已确认：

- 当前版本允许手动直接配置 AI 与数据源。
- 后续若接入账号体系，可演进为按账号权限自动分配。
- 这一未来方向不应反向增加本版本的复杂度。

## 9. 高级与诊断二级页冻结

`高级与诊断` 是面向正式产品的高级能力入口，不是开发控制台。

### 9.1 保留内容

本轮已确认，正式版可保留以下内容：

- 连接检查
- 同步状态
- 扩展入口
- 通知检查
- 自动执行检查
- 维护与清理

### 9.2 自动执行检查

本轮已确认：

- 正式版可保留简化版自动执行检查卡。
- 只显示以下用户可理解信息：
  - 自动执行状态
  - 后台运行状态
  - 最近一次运行时间
  - 重新检查
- 不显示宿主、心跳、队列、本轮创建、本轮执行等内部运行明细。

### 9.3 扩展与同步

本轮已确认：

- `扩展管理` 作为更深层独立页保留。
- `高级与诊断` 中只保留一张 `扩展与同步` 卡。
- 该卡负责：
  - 展示当前同步状态
  - 触发一键同步推荐扩展
  - 提供进入 `扩展管理` 的入口

### 9.4 维护与清理

本轮已确认：

- 危险操作单独放在页面最底部的 `维护与清理` 区。
- 该区与普通诊断项明确分隔。
- 存储清理等不可逆操作必须配套更强的风险提示和确认流程。

### 9.5 正式版隐藏项

本轮已确认，以下内容不应出现在正式版 `高级与诊断` 中：

- 本地测试服务预设
- HTTP 白名单
- 面向联调的开发选项
- 其他仅用于内部测试和研发排障的可见入口

如后续仍需保留这些能力，应通过内部构建、隐藏入口或仅开发环境可见的方式处理，而不是放入正式版用户页面。

## 10. 记忆体系冻结

记忆相关设计虽然仍有部分交互细节待补，但其页面层级、职责边界和首屏顺序已经确认完成，故在此冻结。

### 10.1 记忆页与设置页的边界

本轮已确认：

- `记忆` 是独立一级页。
- `设置` 只管理记忆策略，不管理记忆内容。
- 任何会影响未来推荐、默认任务、理解方式的记忆，都必须可见、可解释、可编辑。

### 10.2 记忆写入规则

本轮已确认：

- 用户明确表达的偏好可直接启用。
- 系统推断出的稳定习惯进入 `待确认`。
- 每日摘要默认生成。
- 每日摘要进入记忆页可浏览，但不自动成为偏好规则。

### 10.3 记忆状态

本轮已确认记忆状态分为：

- 已启用
- 待确认
- 每日摘要
- 已停用

### 10.4 记忆页首页顺序

本轮已确认，记忆页第一屏按以下顺序组织：

1. 顶部摘要区
2. 待确认
3. 已启用
4. 每日摘要
5. 已停用

其中：

- `待确认` 优先于其他内容展示。
- 这符合“任何会影响未来系统行为的候选记忆，都必须优先可见”的原则。

### 10.5 记忆页摘要区

本轮已确认，摘要区至少展示：

- 已启用数量
- 待确认数量
- 今日摘要数量
- 已停用数量

### 10.6 待确认记忆卡

本轮已确认，`待确认` 列表采用紧凑卡片，不在列表中直接铺开完整判断依据。

卡面至少展示：

- 标题
- `待确认` 状态标签
- 来源一句话
- 影响说明一句话
- 更新时间

卡面操作固定为：

- `确认启用`
- `暂不使用`
- `查看理由`

其中：

- `查看理由` 进入独立详情页，不在当前列表直接展开。
- `暂不使用` 不等于删除，而是转入 `已停用`。

### 10.7 待确认记忆详情页

本轮已确认，`待确认` 详情页第一屏顺序固定为：

1. 判断依据与来源
2. 影响说明
3. 记忆正文预览
4. 操作区

操作区固定包含：

- `确认启用`
- `暂不使用`
- `编辑后启用`

确认原则：

- 用户先理解“为什么要记住”，再决定“是否启用”。
- 允许用户修改系统推断出的措辞、范围或粒度后再启用。

### 10.8 已启用与已停用卡片

本轮已确认：

- `已启用` 记忆卡操作为：
  - 主操作：`查看详情`
  - 次操作：`停用`
- `已停用` 记忆卡操作为：
  - 主操作：`重新启用`
  - 次操作：`查看详情`

这样三种主要状态形成明确状态机：

- `待确认` -> `确认启用` / `暂不使用`
- `已启用` -> `停用`
- `已停用` -> `重新启用`

### 10.9 每日摘要卡

本轮已确认：

- `每日摘要` 与长期偏好记忆分开管理。
- `每日摘要` 默认不自动成为偏好规则。
- `每日摘要` 列表卡主操作为 `查看详情`。
- 与“生成记忆”相关的动作采用状态感知显示，不做固定按钮文案。

状态感知规则如下：

- 尚未提炼过：显示 `生成记忆`
- 已经提炼过：显示 `查看已生成记忆`
- 部分内容已提炼：显示 `查看提炼结果`

当一张摘要卡中只有部分内容已提炼时，列表卡仍采用 `查看提炼结果`，而不是继续直接显示 `生成记忆`。

### 10.10 每日摘要去重规则

本轮已确认，`每日摘要 -> 生成记忆` 需要具备去重机制，避免重复提炼同一类记忆。

首版去重原则固定为三层：

1. 结构化键去重
2. 来源链路去重
3. 相近内容合并提示

具体含义如下：

- 结构化键去重
  - 按记忆类型、对象、作用范围、时间粒度生成稳定键。
  - 如果结构化键一致，则不重复生成新的长期记忆。
- 来源链路去重
  - 摘要提炼出的记忆应记录其来源摘要和来源片段。
  - 用户重新打开同一份摘要时，系统应能明确告诉用户哪些内容已经提炼为记忆。
- 相近内容合并提示
  - 当不是完全同键，但高度相近时，不直接重复生成。
  - 系统应提示查看现有记忆或合并处理，而不是静默创建重复条目。

### 10.11 每日摘要详情页的提炼结果

本轮已确认，`每日摘要` 详情页需要有明确的提炼结果区，用于解释当前摘要与长期记忆之间的关系。

该区域至少应能表达：

- 已提炼为记忆的内容
- 对应记忆当前所处状态
  - 已启用
  - 待确认
  - 已停用
- 尚未提炼的内容
- 已存在相近记忆时的合并提示

## 11. 文案与正式化规则冻结

本轮已确认，后续用户可见页面统一遵守以下规则：

- 使用正式产品语气。
- 去掉开发阶段说明。
- 去掉架构解释。
- 去掉迁移叙事。
- 不直接暴露 runtime、compat、draft bundle 等内部术语。

页面文案只做三件事：

1. 告诉用户这里是什么。
2. 告诉用户当前发生了什么。
3. 告诉用户下一步可以做什么。

## 13. 结论

截至 2026-03-13，本轮已经确认完成并写入文档的核心结论是：

- 产品以对话中心为首页和协作主入口。
- 移动端采用顶部状态条 + 左上角抽屉侧栏。

- 记忆内容与设置彻底分离，记忆页作为独立一级页成立。
